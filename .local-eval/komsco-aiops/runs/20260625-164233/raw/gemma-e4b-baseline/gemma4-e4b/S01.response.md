## OpenShift Pod CrashLoopBackOff 진단 및 조치 가이드

특정 네임스페이스의 Pod가 `CrashLoopBackOff` 상태인 경우, 원인은 다양하며 체계적인 접근이 필요합니다. 아래는 운영자가 즉시 따라 할 수 있는 확인 항목과 조치 순서입니다.

---

### 🔍 1단계: 초기 진단 및 정보 수집 (안전한 확인 작업)

가장 먼저 Pod의 상태와 로그를 확인하여 실패 원인을 파악하는 것이 목표입니다. **이 단계에서는 시스템에 어떠한 변경도 가하지 않습니다.**

**✅ 실행할 `oc` 명령어:**

1. **Pod 상세 상태 및 이벤트 확인 (필수):**
   ```bash
   oc describe pod <pod-name> -n <namespace-name>
   ```
   * **확인 포인트:** `Events` 섹션을 집중적으로 확인합니다. OOMKilled, ImagePullBackOff 등 명시적인 실패 원인이 기록되어 있을 수 있습니다.

2. **최근 로그 확인 (필수):**
   ```bash
   oc logs <pod-name> -n <namespace-name> --previous
   ```
   * **확인 포인트:** `--previous` 옵션을 사용하여 이전 컨테이너의 종료 로그를 확인합니다. 애플리케이션 레벨에서 예외(Exception)가 발생했는지 확인하는 것이 가장 중요합니다.

3. **Pod 정의 및 리소스 확인 (선택적):**
   ```bash
   oc get pod <pod-name> -n <namespace-name> -o yaml
   ```
   * **확인 포인트:** `resources` 섹션을 확인하여 CPU/Memory 제한(Limits)이 너무 낮게 설정되어 Pod가 리소스 부족으로 강제 종료되는지 확인합니다.

---

### 💡 원인 후보 우선순위 (진단 순서)

수집된 정보를 바탕으로 다음 순서로 원인을 추론하고 조치 계획을 세웁니다.

1. **[최우선] 애플리케이션 코드/설정 오류:** Pod가 시작하자마자 비즈니스 로직 오류(예: 필수 환경 변수 누락, 잘못된 설정 파일 경로)로 인해 즉시 종료되는 경우. (로그에서 가장 많이 확인됨)
2. **[높음] 리소스 부족 또는 제한 초과:** 컨테이너가 할당된 메모리나 CPU를 초과하여 Kubelet/OpenShift에 의해 강제 종료(OOMKilled 등)되는 경우. (`describe` 이벤트 로그에서 확인 가능)
3. **[중간] 이미지 문제 또는 의존성 오류:** 사용하려는 Docker Image 자체가 존재하지 않거나, 접근 권한이 없거나, 빌드 과정에서 문제가 발생한 경우. (ImagePullBackOff/ErrImagePull 등)
4. **[낮음] 네트워크 또는 서비스 종속성 문제:** Pod가 시작하는 데 필요한 외부 서비스(DB, 캐시 서버 등)에 연결하지 못해 초기화 단계에서 실패하는 경우.

---

### 🛠️ 2단계: 조치 및 해결 (위험도 고려)

원인 분석 결과에 따라 다음 단계를 수행합니다. **반드시 원인을 파악한 후에만 조치를 취해야 합니다.**

#### A. [가장 안전] 환경 변수/설정값 수정 (운영자 권장 첫 시도)
* **상황:** 로그에서 특정 설정 파일이나 환경 변수가 누락되었다는 징후가 보일 때.
* **조치:** `DeploymentConfig` 또는 `Deployment`의 YAML을 수정하여 필요한 환경 변수나 설정을 추가합니다.
    ```bash
    # 예시: Deployment를 편집하고 변경 사항 적용 후, Pod 재시작 유도 (롤아웃)
    oc set env <deployment-name> KEY=VALUE -n <namespace-name> 
    # 또는
    oc rollout restart deployment/<deployment-name> -n <namespace-name>
    ```

#### B. [중간 위험] 리소스 증설 (필요시)
* **상황:** `describe` 이벤트에서 메모리 부족(OOMKilled)이 확인되었을 때.
