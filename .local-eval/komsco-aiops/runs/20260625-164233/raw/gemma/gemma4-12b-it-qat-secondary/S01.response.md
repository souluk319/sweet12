OpenShift 클러스터에서 Pod가 `CrashLoopBackOff` 상태인 경우, 이는 컨테이너가 시작된 후 비정상 종료되어 재시작을 반복하고 있음을 의미합니다. 운영자는 아래의 단계에 따라 원인을 파악하고 조치해야 합니다.

### 1단계: 기초 정보 수집 (안전한 확인 작업)
가장 먼저 Pod의 상태와 로그를 통해 직접적인 에러 메시지를 확인합니다.

*   **실행 명령:**
    ```bash
    # 1. Pod 상세 상태 및 이벤트 확인 (이벤트 섹션에서 'Warning' 확인)
    oc describe pod <pod_name> -n <namespace>

    # 2. 컨테이너 로그 확인 (최근 재시작 전의 로그 추출)
    oc logs <pod_name> -n <namespace> --previous
    ```
*   **확인 포인트:**
    *   `Events`: `OOMKilled`, `Liveness probe failed`, `BackOff` 메시지 확인.
    *   `Logs (--previous)`: 애플리케이션 내부의 Stack Trace, DB 연결 실패, 설정 파일 누락 여부 확인.

---

### 2단계: 원인 후보 우선순위 (RCA 분석)
로그와 Describe 결과를 바탕으로 다음 순서로 원인을 추정합니다.

1.  **애플리케이션 오류 (Application Error):** 코드 버그, 잘못된 환경 변수, DB 연결 실패 등 (가장 빈번).
2.  **리소스 부족 (Resource Exhaustion):** 메모리 초과(`OOMKilled`) 또는 CPU 제한으로 인한 응답 지연.
3.  **프로브 실패 (Probe Failure):** Liveness/Readiness Probe 설정이 잘못되어 컨테이너를 강제로 재시작시키는 경우.
4.  **권한 및 네트워크 문제:** ServiceAccount 권한 부족, Secret/ConfigMap 마운트 실패, 방화벽/NetworkPolicy 차단.

---

### 3단계: 조치 순서 (Operator Action Plan)

#### [순서 1] 환경 설정 검증 (안전함)
*   **작업:** `oc get configmap`, `oc get secret` 명령으로 애플리케이션에 필요한 설정이 정확히 주입되었는지 확인합니다.
*   **조치:** 잘못된 환경 변수나 Secret 값이 있다면 수정 후 반영합니다.

#### [순서 2] 리소스 할당량 검토 (안전함)
*   **작업:** `oc describe pod`에서 `Last State: Terminated`의 `Reason`이 `OOMKilled`인지 확인합니다.
*   **조치:** 필요 시 Deployment/StatefulSet의 `resources.limits.memory`를 상향 조정합니다.

#### [순서 3] 프로브 설정 임시 완화 (주의 요망)
*   **작업:** 애플리케이션 부팅 시간이 길어 Liveness Probe가 먼저 실패하는 경우입니다.
*   **조치:** `initialDelaySeconds`를 늘리거나, 테스트 기간 동안 해당 프로브를 일시적으로 비활성화합니다.

#### [순서 4] 컨테이너 재시작 및 강제 배포 (위험성 있음)
*   **작업:** 설정 변경이 반영되지 않거나 상태가 꼬인 경우 수행합니다.
*   **명령어:** `oc rollout restart deployment <deployment_name> -n <namespace>`
*   **위험 노트:** 서비스 중단(Downtime)이 발생할 수 있으므로, 반드시 트래픽 분산이 가능한 환경에서 실행해야 합니다.

---

### ⚠️ 운영자 주의사항 (Risk Notes)
1.  **`oc delete pod` 남용 금지:** Pod를 삭제하면 ReplicaSet에 의해 즉시 재생성되지만, 근본 원인이 해결되지 않으면 다시 `CrashLoopBackOff`에 빠집니다. 로그 확인이 우선입니다.
2.  **리소스 과다 할당 주의:** `OOMKilled`를 막기 위해 리소스를 무제한으로 늘리면 노드 전체의 자원 고갈을 초래할 수 있습니다. 모니터링 지표(Prometheus/Grafana)를 기반으로 조정하십시오.
3.  **ConfigMap/Secret 변경 시:** 변경 사항이 즉시 Pod에 반영되지 않을 수 있습니다. `rollout restart`가 필요함을 인지해야 합니다.
