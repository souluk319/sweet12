제공된 지침에 따라 RHEL, Windows Server, OpenShift Pod 환경에서 CPU 사용률 급증 원인을 조사하기 위한 초기 확인 명령을 정리하였습니다.

### 1. RHEL (Red Hat Enterprise Linux) 서버
RHEL 환경에서는 시스템 전체의 리소스 할당과 프로세스별 점유율을 파악하는 것이 우선입니다.

| 순서 | 실행 명령 | 선택 이유 | 위험/부하 여부 |
| :--- | :--- | :--- | :--- |
| 1 | `top` 또는 `htop` | 실시간 CPU 사용량, 로드 에버리지(Load Average), 상위 프로세스 목록을 즉시 파악하기 위함 | 일반 (htop은 설치 필요) |
| 2 | `vmstat 1 5` | 시스템 전체의 상태(Context Switch, Interrupt, I/O Wait 등)를 요약하여 병목 지점 확인 | 안전 |
| 3 | `iostat -xz 1` | CPU 사용량 중 I/O Wait이 높은 경우 디스크 성능 문제인지 확인하기 위함 | 안전 |
| 4 | `sar -u 1 5` | 특정 시점의 CPU 사용률 추이를 분석하여 급증 패턴 파악 (Historical data) | 안전 |
| 5 | `lsof -i :<port>` | 특정 서비스 포트에서 과도한 연결이 발생하여 CPU를 점유하는지 확인 | 안전 |

---

### 2. Windows Server
Windows 환경에서는 작업 관리자와 성능 모니터링 도구를 통해 프로세스 우선순위와 스레드 상태를 확인합니다.

| 순서 | 실행 명령 (PowerShell/CLI) | 선택 이유 | 위험/부하 여부 |
| :--- | :--- | :--- | :--- |
| 1 | `Get-Process | Sort-Object CPU -Descending | Select-Object -First 10` | 현재 CPU를 가장 많이 사용하는 상위 10개 프로세스 식별 | 안전 |
| 2 | `typeperf "\Processor(_Total)\% Processor Time" -si 5` | 실시간으로 전체 프로세서의 사용률을 수치화하여 모니터링 | 안전 |
| 3 | `Get-Counter '\Process(*)\% Processor Time' | Sort-Object CounterValue -Descending | Select-Object -First 10` | 개별 프로세스의 스레드 단위 CPU 점유율 상세 확인 | 안전 |
| 4 | `netstat -an | findstr ESTABLISHED` | 네트워크 연결 과부하로 인한 CPU 인터럽트 발생 여부 간접 확인 | 일반 (연결이 많을 경우 출력 지연) |
| 5 | `Get-WmiObject Win32_PerfRawData_PerfProc_Processor` | 하드웨어 수준의 상세 성능 카운터 수집 (고급 분석용) | 안전 |

---

### 3. OpenShift Pod
OpenShift 환경에서는 컨테이너 리소스 제한(Limit)과 노드 수준의 자원 경합을 동시에 확인해야 합니다.

| 순서 | 실행 명령 (`oc` CLI) | 선택 이유 | 위험/부하 여부 |
| :--- | :--- | :--- | :--- |
| 1 | `oc top pod -A --sort-by=cpu` | 클러스터 내 모든 Pod 중 CPU를 과도하게 사용하는 대상 식별 | 안전 |
| 2 | `oc describe pod <pod_name>` | 해당 Pod의 Resource Limits, Requests 및 Events(OOMKilled 등) 확인 | 안전 |
| 3 | `oc logs <pod_name> --tail=100` | 애플리케이션 로그에서 에러 루프나 과도한 요청 처리 기록 확인 | 안전 |
| 4 | `oc adm top node` | 특정 Pod이 아닌 노드 자체의 자원 고갈(Noisy Neighbor) 여부 확인 | 안전 |
| 5 | `oc debug node/<node_name>` | (필요 시) 노드에 직접 접속하여 커널 레벨의 프로세스 분석 수행 | **위험** (노드 접근 권한 및 영향도 고려 필요) |

---

### [Operator-Ready RCA 구조 가이드]
원인 파악 후 보고서 작성을 위해 다음 구조를 활용하십시오.

1.  **Symptoms (증상):** CPU 사용률 수치, 발생 시점, 영향받은 서비스 범위.
2.  **Evidence (증거):** 위 명령들을 통해 수집된 상위 프로세스 ID(PID), I/O Wait 수치, 로그 메시지.
3.  **Root Cause (근본 원인):** 
    *   *Application:* 무한 루프, 비효율적인 알고리즘, 메모리 누수로 인한 GC 과부하.
    *   *Infrastructure:* 디스크 I/O 병목, 네트워크 인터럽트 폭증, 노드 자원 경합.
4.  **Action Taken (조치 사항):** 프로세스 재시작, 리소스 Limit 조정, 코드 패치 등.
5.  **Prevention (재발 방지):** 알람 임계치 조정, HPA(Horizontal Pod Autoscaler) 설정 최적화.
