## Root Cause Analysis (RCA) Report

**보고서 개요**
*   **대상 서비스:** `payment-api` (Namespace: `payment`)
*   **현상:** Pod가 `CrashLoopBackOff` 상태로 반복 재시작됨.
*   **영향 범위:** 결제 관련 API 서비스 중단.

---

### 1. 현상 분석 (Symptoms)
*   `payment-api` Pod가 실행되지 못하고 지속적으로 재시작됨 (`CrashLoopBackOff`).
*   이전 로그 확인 결과, PostgreSQL 데이터베이스 연결 시 **"FATAL password authentication failed for user 'payment_app'"** 오류 발생.

### 2. 원인 분석 (Root Cause Analysis)
**가장 가능성 높은 원인: DB 접속 정보(비밀번호) 불일치**
*   `oc rollout history`를 확인한 결과, Revision 13에서 `DB_SECRET_NAME=payment-db-secret`으로 설정이 변경되었습니다.
*   현재 Pod는 해당 Secret을 참조하여 DB에 접속을 시도하고 있으나, Secret 내에 저장된 `payment_app` 사용자의 비밀번호가 실제 PostgreSQL 데이터베이스에 등록된 비밀번호와 일치하지 않습니다.

**[추정 사항]**
*   Secret이 생성될 때 잘못된 비밀번호가 입력되었거나, DB의 비밀번호는 변경되었으나 Secret은 업데이트되지 않았을 가능성이 높습니다.
*   네트워크 연결 자체의 문제는 아니며(Host 이름 및 포트 도달은 성공), 인증 단계에서 거부되고 있습니다.

### 3. 추가 확인 명령 (Verification Commands)
정확한 원인 파악을 위해 아래 명령어를 실행하여 Secret 내용과 실제 DB 상태를 대조해야 합니다.

*   **Secret 내 비밀번호 확인:**
    ```bash
    oc get secret payment-db-secret -n payment -o jsonpath='{.data.password}' | base64 --decode
    ```
*   **DB 접속 테스트 (임시 Pod 활용):**
    (Secret의 정보를 사용하여 직접 DB에 접속 가능한지 확인)
    ```bash
    oc run db-test --image=postgres:latest -n payment --env="PGPASSWORD=$(oc get secret payment-db-secret -o jsonpath='{.data.password}' | base64 --decode)" -- env PGHOST=postgres.payment.svc PGUSER=payment_app psql -h postgres.payment.svc -p 5432 -d [DB_NAME]
    ```

### 4. 즉시 조치 사항 (Immediate Actions)
*   **Secret 업데이트:** 올바른 비밀번호로 `payment-db-secret`을 수정합니다.
    ```bash
    oc create secret generic payment-db-secret \
      --from-literal=password=<Correct_Password> \
      -n payment --dry-run=client -o yaml | oc apply -f -
    ```
*   **Pod 재시작:** Secret 수정 후 Deployment를 롤아웃하여 반영합니다.
    ```bash
    oc rollout restart deploy/payment-api -n payment
    ```

### 5. 재발 방지책 (Preventive Measures)
*   **Secret 동기화 프로세스 수립:** DB 비밀번호 변경 시 Secret을 동시에 업데이트하는 표준 운영 절차(SOP)를 마련합니다.
*   **ConfigMap/Secret 검증 자동화:** 배포 파이프라인(CI/CD)에 접속 테스트 단계를 추가하여, 애플리케이션이 실행되기 전 DB 연결성을 확인하는 'Readiness Check' 또는 'Smoke Test'를 도입합니다.
*   **Secret 관리 도구 활용:** 수동 Secret 관리를 지양하고, HashiCorp Vault와 같은 외부 Secret Manager를 연동하여 동기화 오류를 최소화합니다.
